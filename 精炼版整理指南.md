# 音频转录精炼版整理指南（LLM Prompt 模板）

> 本指南用于指导大模型将 ASR 原始转录整理为高质量精炼版，适用于会议录音、访谈、播客等场景。
> 版本：v2.0 | 基于 50+ 篇精炼版实战经验提炼

---

## 一、核心原则

1. **保留论证过程，不止提炼结论**——用户需要掌握详细信息，不是只要 bullet points
2. **信息密度优先**——表格/图形是为了更高信息密度和更好突出信息，不为做而做
3. **时间锚定**——每个章节标注原始音频时间区间，便于回溯定位
4. **完整性可验证**——精炼版必须覆盖原始转录 100% 的内容区间

---

## 二、文头元信息（必须）

在精炼版开头写明：

```markdown
> *转录模型：XXX | 转录时间：YYYY-MM-DD | 设备：XXX*
> *音频时长：XX分XX秒 | 转录段数：XX段 | 完整性验证：✅ 通过（XX/XX段，X%失败，覆盖00:00-XX:XX）*

> *录音来源：XXX | 文件：XXX | 大小：XXX*
> *录音性质：XXX | 参会方/嘉宾：XXX*
```

- 元信息分两块：技术元信息（转录相关）+ 来源元信息（录音相关）
- 完整性验证状态在整理完成、验证通过后更新为 ✅ 通过

---

## 三、章节划分逻辑

### 3.1 划分方法

- **按话题切换划分**，不按等时间切
- 一个话题可能横跨多个不连续的时间段——用主时间区间标注即可
- 章节编号用中文数字（一、二、三...）
- 每个章节标题后标注时间区间：`## 章节名 【MM:SS-MM:SS】`

### 3.2 命名规则

- 标题应**概括章节核心议题**，而非简单重复原话
- 如果一个大话题下有多个子议题，用 `###` 子标题拆分
- 子标题用编号（1.1, 1.2...），便于引用

### 3.3 章节粒度

- 一段 30 分钟的录音，通常产出 **8-15 个章节**
- 单个章节内容不宜超过全文 15%，否则说明拆分不够细
- 开头寒暄/结尾客套可略写或合并，但**不可省略**

---

## 四、内容整理手法

### 4.1 何时用表格

| 适合用表格 | 不适合用表格 |
|-----------|-------------|
| 多维度对比（如 A/B/C 方案对比） | 一句话能说清的事实 |
| 结构化数据（注册资本、人员名单） | 叙述性/论证性内容 |
| 时间线/里程碑 | 单一数值 |
| 多方观点并列 | 情感/语气描述 |

**表格设计原则**：
- 列名用中文，简洁明确
- 同一张表不超过 6 列
- 如果一个表格超过 8 行，考虑拆分或改为多个小表

### 4.2 保留原话 vs 改写

- **关键定义/论断/决策**：保留原话（用引号或直接引用风格）
- **背景叙述/过程描述**：可改写为精炼语句
- **数字/金额/时间节点**：必须精确保留，不得模糊化
- **逻辑推导过程**：保留因果链，不能只给结论

### 4.3 处理口语冗余

ASR 转录的口语特征需要清理：

| 口语特征 | 处理方式 |
|---------|---------|
| 重复词（"然后然后然后"） | 保留一次 |
| 口头禅（"嗯嗯"、"对对对"） | 删除，除非表达确认/同意语义 |
| 自我纠正（"不是A，是B"） | 只保留最终结论 B |
| 模糊表达（"大概两三千万"） | 原样保留，不擅自精确化 |
| 未完成句子 | 保留意图，补全语义 |
| ASR 错误（同音字等） | 根据上下文修正，明显错误直接改 |

### 4.4 信息分层

对每个章节，按以下优先级组织信息：

1. **核心结论/决策**（最重要的放最前面）
2. **支撑论据/过程**（为什么得出这个结论）
3. **具体数据**（金额、人数、时间节点）
4. **背景/上下文**（帮助理解的信息）
5. **待定/未决事项**（如有）

### 4.5 高级技巧

#### 4.5.1 因果链保留法

**❌ 错误**：只写结论
```
华瑞有税务问题，已解决。
```

**✅ 正确**：保留完整因果链
```
**实际路径**：
- 华瑞航空是皮拉图斯大中华区唯一经销商
- 皮拉图斯账上留存大量未分配的经销商佣金
- 华瑞指示皮拉图斯不将佣金汇入中国，而直接付给境外各项研发费用

**税务问题**：
- 按中国税法，应先确认为华瑞中国收入，再付给境外
- 实际未走此路径，构成历史遗留税务问题

**解决方案**：
- 不额外提供书面依据，视为"境外创始股东通过其他方式的投入"
```

#### 4.5.2 模糊表达处理

口语中常见的模糊数字**必须原样保留**，不能擅自精确化：

```
"大概两三千万"  → 保留"两三千万"，不改为"3000万"或"2000-3000万"
"刚刚过百分之五十"  → 保留"刚刚过百分之五十"，不改为"51%"
"一年多"  → 保留"一年多"，不改为"1年+"或"13-14个月"
```

#### 4.5.3 多方对话标注

当录音涉及多方对话时：

```markdown
**张伟（创始人）**：我们准备在5月初开TCB会议...

**李律师（投资方顾问）**：这个时间点你们有把握吗？复材验证的进度如何？

**张伟**：三线并进，复材验证+民航研究院验证+西工大咨询...
```

- 首次出现时标注角色
- 后续可用简称
- 不同方观点用加粗区分

## 五、特殊内容类型处理

### 5.1 会议录音

- 开头标注参会方角色（谁代表哪方）
- 明确区分"已决定"vs"待讨论"vs"存在分歧"
- 敏感话题（税务/合规等）如实记录，不加评判

### 5.2 访谈/播客

- 区分主持人提问和嘉宾回答
- 嘉宾的核心观点用加粗或独立段落突出
- 主持人串联词可精简，但提问意图要保留

### 5.3 技术讲解

- 专有名词保留原文，必要时加中文注释
- 技术架构/流程用嵌套列表或表格
- 不确定的技术细节标注 `[待确认]`

### 5.4 电话会议/远程尽调

- 标注通话性质（尽调/商务谈判/项目汇报）
- 区分各方立场和利益诉求
- 争议点用对比表格展示各方观点
- 承诺/约定事项单独列出

## 六、文末速览表（推荐）

在精炼版末尾添加**关键数据速览表**，提取全文最核心的数字/结论：

```markdown
## 关键数据速览

| 指标 | 数值 |
|------|------|
| XXX | XXX |
```

- 只放**最关键的 5-10 个数据点**
- 让读者 10 秒内抓住全文要点
- 数据点应是全文最常被引用/查阅的信息

---

## 七、完整性验证流程

精炼版完成后，必须执行验证：

1. **段数验证**：原始转录的段数 vs 精炼版覆盖的时间区间
2. **时间覆盖**：精炼版最早时间标注 ≈ 00:00，最晚时间标注 ≈ 音频时长
3. **尾部检查**：原始转录最后 5-10 段的内容是否在精炼版中有对应
4. **遗漏检查**：如果原始转录有 N 个主要话题切换，精炼版应有对应 N±2 个章节

验证结果更新到文头元信息的"完整性验证"字段。

### 验证脚本示例

```python
import re

# 读取原始转录
content = open('原始转录.txt', 'r', encoding='utf-8').read()
lines = content.strip().split('\n')
times = re.findall(r'\[(\d+:\d{2}(?::\d{2})?)\]', content)
failed = sum(1 for l in lines if '转录失败' in l)
print(f'原始转录: {len(lines)}段, 时间范围 {times[0]}-{times[-1]}, 失败{failed}段')

# 读取精炼版
refined = open('精炼版.md', 'r', encoding='utf-8').read()
refined_times = re.findall(r'【(\d+:\d{2}(?::\d{2})?)', refined)
print(f'精炼版时间标注: {len(refined_times)}个区间, {refined_times[0]}-{refined_times[-1]}')

# 验证覆盖率
if times and refined_times:
    orig_end = times[-1]
    refined_end = refined_times[-1]
    print(f'覆盖检查: 原始结束于 {orig_end}, 精炼版结束于 {refined_end}')
```

## 八、输出规范

| 项目 | 要求 |
|------|------|
| 格式 | Markdown (.md) |
| 文件名 | `{主题}_精炼版.md` |
| 编码 | UTF-8 |
| 同时交付 | 精炼版 + 原始转录（两个 md 文件） |
| 原始转录 | 直接将 .txt 重命名为 .md，不做修改 |

## 九、质量自检清单

整理完成后，逐项检查：

- [ ] 文头元信息完整（转录模型、时长、段数、验证状态）
- [ ] 来源元信息完整（录音来源、性质、关键人物）
- [ ] 每个章节有时间锚定
- [ ] 表格使用合理（对比/结构化数据，非滥用）
- [ ] 关键数字精确保留
- [ ] 论证过程保留（非仅结论）
- [ ] 口语冗余已清理
- [ ] 模糊表达未擅自精确化
- [ ] 尾部内容未遗漏
- [ ] 完整性验证通过
- [ ] 关键数据速览表已添加

## 十、示例 Prompt

将以上指南浓缩为可直接喂给其他 LLM 的 prompt：

```
你是一个专业的音频转录精炼版整理专家。请将以下 ASR 原始转录整理为精炼版。

## 整理规则

1. **文头**：写明转录模型、时长、段数、完整性验证状态；以及录音来源、性质、关键人物
2. **章节**：按话题切换划分，每章标注时间区间【MM:SS-MM:SS】，中文数字编号
3. **内容**：保留论证过程和关键原话，不只提炼结论；数字必须精确；清理口语冗余（重复词/口头禅/自我纠正）
4. **表格**：对比、结构化数据、多维度并列时用表格；一句话能说清的不要硬做表格
5. **信息分层**：核心结论 → 支撑论据 → 具体数据 → 背景上下文 → 待定事项
6. **模糊表达**：原样保留（如"大概两三千万"），不擅自精确化
7. **因果链**：保留完整推理过程，不只写结论
8. **多方对话**：首次出现标注角色，不同方观点用加粗区分
9. **文末**：添加关键数据速览表（5-10个核心数据点）
10. **验证**：完成后核对原始转录段数与精炼版时间覆盖，确认无遗漏

现在请整理以下原始转录：

{原始转录内容}
```

## 十一、常见错误与修正

| 错误 | 正确做法 |
|------|---------|
| 只写"公司有税务问题" | 写清原因→过程→解决方案的完整因果链 |
| 表格里只有1行数据 | 一行数据用列表或正文，不用表格 |
| "大概3000万"（擅自精确） | 保留原话"大概两三千万" |
| 章节按5分钟等时切分 | 按话题切换划分，允许时长不等 |
| 省略开头寒暄 | 合并略写，但不可省略 |
| 遗漏尾部5分钟 | 必须检查最后5-10段内容是否覆盖 |
